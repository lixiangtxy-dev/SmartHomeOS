## 1. 内核层 (OS_Core)
定位：内核层在资源受限的MCU上实现支持优先级抢占式的实时操作系统内核。通过控制块的高度抽象，使内核核心逻辑与上层智能家居业务完全解耦。

### 1.1 核心模块技术实现拆解

#### A. 任务管理与多任务抢占式调度(task.c & switch.S)

1. 任务控制块 (TCB, Task Control Block)：设计了 TCB 结构体，完整记录了任务的当前栈指针（SP）、任务状态（就绪、运行、阻塞、挂起）、任务优先级以及延时节拍数。

2. 硬件级上下文切换 (switch.S)：在 RISC-V 架构下，通过纯汇编代码实现了任务上下文的保存与恢复。在调度发生时，将当前任务的通用寄存器（x1/ra, x5-x31 等）压入当前任务栈，随后切换 CPU 的 sp 寄存器到新任务栈，并弹出新任务的寄存器环境。

3. 抢占式调度策略：构建了基于优先级的就绪任务列表。在每次系统时钟节拍中断（SysTick）中，内核会递减阻塞任务的延时计数器，并重新评估当前运行任务与就绪队列中最高优先级任务的优先级。一旦发现更高优先级任务，立即触发软件中断，调用 switch.S 进行强制抢占。

#### B.时间确定性动态内存池 (mempool.c)
为了规避标准 C 库 malloc/free 带来的内存碎片化以及执行时间不可控（非确定性）的问题：
1. 固定块内存池 (Memory Pool)：内核划分了连续的静态内存区域，并将其切分为固定大小的内存块，通过链表进行管理。

2. $O(1)$ 时间复杂度：内存的申请与释放均只需对链表表头进行单次操作，执行时间完全恒定，极大地提升了系统的实时响应速度，满足嵌入式场景的安全性要求。

#### C.多任务同步与异步通信 (mutex.c & msgqueue.c)

在处理智能家居网络接收、传感器采集等并发业务时，内核提供了两种高效的 IPC 机制：
1. 互斥锁 (Mutex)：用于保护共享外设资源（如 SPI 总线、OLED 屏幕）。当资源被占用时，请求任务会被移出就绪列表并挂载到该锁的阻塞队列中，让出 CPU 使用权。

2. 异步消息队列 (Message Queue)：基于环形缓冲区（Ring Buffer）实现。当网络驱动收到云端控制指令时，通过队列异步投递给应用层任务，实现了“生产者-消费者”模型的低耦合架构。

## 2. 硬件抽象层 (OS_HAL)

OS_HAL位于操作系统内核 (OS_Core) 与具体芯片固件库 (MCU_Support) 之间，设计目的是隔离硬件差异，为上层驱动提供统一、标准的接口。

目前鉴于项目规模需求，我只封装三个硬件总线：GPIO，SPI，ADC

### 1. GPIO 抽象 (hal_gpio.c / .h)
将复杂的寄存器配置抽象为最基础的引脚操作：
1. 统一初始化：屏蔽了底层时钟使能、模式配置（推挽输出、浮空输入等）的繁琐细节。

2. 标准 API：例如封装了类似 HAL_GPIO_WritePin(port, pin, state) 和 HAL_GPIO_ReadPin(port, pin) 的标准函数，供上层 LED、继电器或按键驱动直接调用。

### 2. SPI 接口抽象 (hal_spi.c / .h)
SPI 是智能家居设备中连接高速外设的主力总线。

1. 传输封装：实现了标准的单字节收发、多字节块读写接口。

2. 片选控制：规范了硬件 CS（Chip Select）引脚的拉高拉低时序控制逻辑。

3. 同步与互斥预留：结合 OS_Core 的互斥锁（Mutex）机制，在 hal_spi 层预留或集成了并发保护，确保多任务同时请求 SPI 总线时不会发生数据错乱。

### 3. ADC 模数转换抽象 (hal_adc.c / .h)
用于连接环境传感器（如光敏电阻 LDR）。

1. 通道抽象：将底层的引脚与 ADC 通道映射关系进行封装。

2. 数据采集：提供单次采样和多次采样求平均值的标准接口，向上层屏蔽了底层 ADC 的校准、触发和等待转换完成的硬件细节。


## 3. 硬件设备驱动层 (OS_Drivers)

OS_Drivers 这一层基于底层的 OS_HAL（硬件抽象层）接口，构建了具体外设组件的控制逻辑。它不仅负责驱动物理硬件，还包含了实现智能家居场景所需的各种协议栈和中间件适配。

### A.存储系统集成与适配 (drv_w25q80.c & fs_port.c)

智能家居设备通常需要非易失性存储来保存配置信息、日志或离线数据。

1. W25Q80 SPI Flash 驱动 (drv_w25q80.c)：实现了对外部 W25Q80 存储芯片的基本读、写、擦除操作，包括各种粒度（扇区、块、整个芯片）的擦除以及状态寄存器的读取，确保数据写入的安全性和完整性。

2. LittleFS 移植接口 (fs_port.c)：这不仅是一个驱动，更是一个适配层。它将底层的 W25Q80 块读写接口转换并挂载给第三方的 LittleFS（轻量级掉电安全文件系统）。这赋予了系统在 Flash 上使用标准文件操作（类似于 POSIX open, read, write）的能力，极大地方便了上层应用进行数据管理。

### B. 可视化模块 (drv_oled.c & oled.c & oledfont.h)

1. OLED 驱动抽象：底层基于 I2C/SPI（依赖 HAL 层配置）实现了屏幕初始化、清屏、像素点亮等基础操作。

2. 字库与图形支持：集成了 ASCII 字库（oledfont.h），支持在屏幕上输出字符、字符串，并提供了基础的几何图形绘制功能，方便展示网络状态、传感器数值等信息。

### C. 物联网通信基础架构 (drv_at.c & drv_mqtt.c)
赋予了设备“上云”的核心能力，这是智能家居 OS 的关键组成部分。

1. AT 指令解析框架 (drv_at.c)：为了适配常见的 WiFi 模组（如 ESP8266/ESP-01S 等），编写了健壮的 AT 指令收发引擎。它通过操作 HAL 层的 UART，实现了发送命令、异步接收响应、处理超时与重发机制，将不稳定的串口通信封装成了可靠的网络层接口。

2. MQTT 协议栈适配 (drv_mqtt.c)：基于底层的网络连接通道，实现了 MQTT协议报文的封装与解析。它负责建立与云服务器的连接，处理主题的发布（Publish）与订阅（Subscribe），使得传感器数据可以上报云端，同时能实时接收云端的控制指令。
